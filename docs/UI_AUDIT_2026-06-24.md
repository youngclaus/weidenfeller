# Portfolio UI Audit

**Repository:** `youngclaus/weidenfeller`  
**Base branch:** `master`  
**Audit branch:** `ui-audit-2026-06-24`  
**Audit date:** 2026-06-24

## Scope and limitations

This audit covers the current source for the shared application shell and the Home, Projects, Explore/About, Music, inventory, theme, and project-modal experiences.

The findings below are source-confirmed unless marked **Runtime verification needed**. A full local build and browser run could not be performed from the audit workspace because outbound network access was unavailable, so `npm ci`, device screenshots, Lighthouse, axe, and real-browser interaction checks remain follow-up work.

## Executive summary

The portfolio has a memorable visual identity and several strong interaction concepts, especially the vinyl navigation, album-style home player, project-card presentation, and collectible Explore room. The main risks are not aesthetic; they are architecture, accessibility, responsive behavior, theme reliability, and production polish.

Recommended order of work:

1. Repair routing and build confidence.
2. Make all primary interactions keyboard and touch accessible.
3. stabilize mobile layouts, especially the header and inventory.
4. Normalize theme tokens and validate contrast.
5. Improve modal behavior, motion preferences, metadata, and content polish.

## Severity definitions

- **P0 — Release blocker:** prevents a reliable production release or makes core pages unreachable.
- **P1 — High:** core functionality, accessibility, or common viewport failure.
- **P2 — Medium:** meaningful usability, performance, maintainability, or polish issue.
- **P3 — Low:** cleanup or consistency improvement.

# P0 — Release blockers

## 1. Next.js routing is bypassed by local component state

**Files:** `pages/_app.tsx`, `pages/index.tsx`, `pages/projects.tsx`, `pages/about.tsx`, `pages/music.tsx`

The custom App imports every page directly and chooses one with `activeComponent` state instead of rendering Next.js's `Component` and `pageProps`.

Consequences:

- `/projects`, `/about`, and `/music` do not reliably represent their corresponding pages.
- Refreshing a secondary page returns the user to Home.
- Browser Back and Forward do not track navigation.
- Deep links cannot be shared.
- Page-specific metadata and analytics are unavailable.
- Every route is effectively a single in-memory view switch.

**Recommendation:** Use actual Next.js routes with `next/link` or `router.push`. Let `_app.tsx` render `<Component {...pageProps} />`. Keep only shared providers and layout in `_app.tsx`.

## 2. No automated build, lint, accessibility, or regression gate

**File:** `package.json`

Only `build`, `dev`, `start`, and `vercel-build` scripts exist. There is no lint, typecheck, unit test, browser test, or accessibility test command.

**Recommendation:** Add at minimum:

- `typecheck`: `tsc --noEmit`
- `lint`: ESLint
- `test`: component/unit tests
- `test:e2e`: Playwright
- `test:a11y`: axe integration
- CI that runs install, typecheck, lint, build, and a smoke test on pull requests

## 3. Header mobile CSS contains an unclosed media block

**File:** `components/Header.tsx`, `VinylText`

The `@media (max-width: 550px)` block closes the styled-components template before closing the CSS brace. This can cause malformed generated CSS or dropped rules.

**Recommendation:** Close the media block explicitly and require CSS/build validation in CI.

# P1 — High-priority findings

## Shared navigation and shell

### 4. Primary navigation is not keyboard accessible

**File:** `components/Header.tsx`

Each vinyl is a clickable `div`. It cannot be reached with Tab and has no button/link semantics, current-page announcement, or visible focus style.

**Recommendation:** Use links for page navigation and a button for theme switching. Add `aria-current="page"`, descriptive accessible names, and `:focus-visible` styling.

### 5. Mobile header is likely clipped at narrow widths

**File:** `components/Header.tsx`

Five overlapping 96px vinyl controls still require more horizontal space than a 300–375px viewport. Global horizontal overflow is hidden, so clipping is masked rather than solved. Text drops to 9px.

**Recommendation:** Replace the full vinyl row with a compact mobile navigation pattern below a breakpoint, or provide a horizontally scrollable, snap-aligned control strip with safe edge padding.

### 6. Theme switch label describes state, not action

**File:** `components/Header.tsx`

The theme control displays `light`, `dark`, or `custom`, but clicking it changes the theme. A user cannot tell whether the label is the current state or the action that will occur.

**Recommendation:** Use an explicit label such as `Switch to dark mode`, expose pressed/state semantics, and include an icon plus visually hidden text.

### 7. No skip link or consistent main landmark

The app has a fixed header and animated content, but pages do not consistently expose a `<main>` landmark or a skip-to-content control.

**Recommendation:** Add a shared layout with `<header>`, `<nav>`, and `<main id="main-content">`, plus a skip link.

### 8. No reduced-motion behavior

**Files:** `pages/projects.tsx`, `components/Header.tsx`, `components/Index/Footer.tsx`, `components/Player/Player.tsx`, Explore components

The site uses infinite gradients, typing effects, floating text, rotations, hover transitions, and animated controls without a `prefers-reduced-motion` override.

**Recommendation:** Disable nonessential animation and smooth scrolling when reduced motion is requested.

## Home

### 9. Player can overflow a 300px viewport

**Files:** `pages/index.tsx`, `components/Player/Player.tsx`

The player has `max-width: 300px` plus 20px padding on both sides with the default content-box model, producing a 340px rendered width. The global 300px minimum viewport and hidden overflow can clip it.

**Recommendation:** Apply `box-sizing: border-box`, use `width: min(100% - 32px, 340px)`, and test at 320px and 300px.

### 10. Short viewport behavior replaces real content with a joke

**File:** `components/Player/Player.tsx`

When height is below 660px, the title and artist become `Break Stuff` and `Limp Bizkit` regardless of the selected slide. This does not solve the layout problem and makes the carousel information inaccurate on common landscape/mobile sizes.

**Recommendation:** Make the component responsive instead of replacing content. Reduce image height, spacing, and control dimensions at short heights.

### 11. Long player text cannot wrap or truncate safely

**File:** `components/Player/Player.tsx`

Song and artist lines use `white-space: nowrap` without overflow handling. Long artist names can leave the card.

**Recommendation:** Allow two-line wrapping or add ellipsis with a title/accessible full label.

### 12. Footer animation accumulates hidden DOM nodes indefinitely

**File:** `components/Index/Footer.tsx`

Every completed typing cycle appends another item to `floatingWords`. The visual item fades, but it is never removed from state or the DOM.

**Recommendation:** Remove each word after its animation completes or render only the current/recent item.

### 13. Home background image is implemented twice

**File:** `pages/index.tsx`

`BackgroundContainer` contains an `<img>` and also sets the same file as a CSS background. This adds redundant markup and can produce unexpected sizing.

**Recommendation:** Keep one implementation. Use CSS for decorative imagery and hide it from assistive technology, or use `next/image` when the image is meaningful.

## Projects

### 14. Project cards are clickable `div` elements

**File:** `components/Projects/ProjectCard.tsx`

Cards cannot be focused or activated with a keyboard and have no button/link role.

**Recommendation:** Use a `<button>` for modal-opening cards or a link for navigational cards. Preserve the card styling and add focus-visible treatment.

### 15. Project modal lacks required dialog behavior

**File:** `components/Projects/ProjectModal.tsx`

Missing behaviors include:

- `role="dialog"` and `aria-modal="true"`
- labelled title/description relationships
- initial focus
- focus trap
- Escape-to-close
- focus return to the triggering card
- background scroll lock/inert content

The image popout has the same issues.

**Recommendation:** Use a tested accessible dialog primitive or implement the complete WAI-ARIA dialog pattern.

### 16. Modal and fixed header share the same z-index

**Files:** `components/Header.tsx`, `components/Projects/ProjectModal.tsx`

Both use `z-index: 100`, making stacking dependent on DOM and stacking context details. The header may visually compete with the modal.

**Recommendation:** Define a small global layer scale, for example content 0, sticky navigation 100, overlay 900, dialog 1000, tooltip 1100.

### 17. Project page has no mobile typography/spacing adjustments

**File:** `pages/projects.tsx`

The page keeps a 4rem title, 40px outer padding, and a large fixed header treatment at all widths.

**Recommendation:** Add responsive title sizing, safe top padding beneath navigation, 16–20px mobile gutters, and smaller grid gaps.

### 18. Styled `active` prop is likely forwarded to the DOM

**File:** `pages/projects.tsx`

`TagButton` uses a non-transient `active` prop. With styled-components this may create a React unknown/non-boolean attribute warning.

**Recommendation:** Rename it to `$active` so it is consumed only by the styled component.

## Explore / About

### 19. Core room interactions are mouse/hover dependent

**File:** `components/About/ImageContainer.tsx`

Descriptions are shown on hover, and unlocked vinyl themes are activated by clicking images. Images are not keyboard focusable and touch users have no equivalent hover discovery.

**Recommendation:** Render each interactive object as a button, support focus and tap, and show descriptions on focus/tap as well as hover.

### 20. Global wheel interception is fragile

**File:** `components/About/ImageContainer.tsx`

A non-passive `wheel` listener is installed on `window`, converting vertical wheel input into horizontal movement and preventing scrolling outside the inventory while open.

Risks:

- surprising page behavior
- trackpad and assistive-device conflicts
- no keyboard equivalent
- event handling beyond the intended container

**Recommendation:** Attach the handler to the scene only, prefer native horizontal scrolling, add visible keyboard controls, and avoid global `preventDefault`.

### 21. Explore layout is hard-coded to one scene geometry

**File:** `components/About/ImageContainer.tsx`

Objects are positioned with large percentage translations such as `translate(1450%, 390%)`. This is difficult to maintain and likely inconsistent across aspect ratios and zoom levels.

**Recommendation:** Use a defined scene canvas coordinate system, aspect-ratio wrapper, and data-driven x/y positions. Add minimap/scroll cues and responsive scaling.

### 22. Inventory controls are not keyboard accessible

**File:** `components/About/InventoryButton.tsx`

The hamburger is a clickable `div`, and `Create`/`Earn` are anchors with no `href`. There are no expanded/menu relationships or focus management.

**Recommendation:** Use buttons and links with `aria-expanded`, `aria-controls`, menu labelling, Escape close, and focus return.

### 23. Inventory has no usable mobile layout

**File:** `components/BlueprintMenu/InventoryManager.tsx`

The inventory always presents three equal columns. At narrow widths, titles, cards, quantities, and controls become compressed. The existing `isMobileView` state is unused.

**Recommendation:** At mobile widths use tabs or a one-column sequence for Stash, Blueprints, and Prints. Keep the series navigator sticky within the modal.

### 24. Series navigator positioning is not safely scoped

**File:** `components/BlueprintMenu/InventoryManager.tsx`

`SeriesNavigator` is absolutely positioned, but the inventory `Container` is not positioned. Its containing block can therefore be an outer overlay rather than the inventory panel itself.

**Recommendation:** Set the inventory container to `position: relative` and verify the scroll region reserves the navigator height.

### 25. Blueprint requirements and print descriptions are hover-only

**Files:** `components/BlueprintMenu/BlueprintComponent.tsx`, `components/BlueprintMenu/PrintsComponent.tsx`

Requirement details and descriptions appear only through hover overlays. Touch and keyboard users cannot reliably access them.

**Recommendation:** Keep essential information visible, or provide an explicit Details toggle that works with touch and keyboard.

### 26. Duplicate-color blueprint costs are calculated incorrectly

**Files:** `components/BlueprintMenu/blueprints.ts`, `components/BlueprintMenu/BlueprintComponent.tsx`, `components/BlueprintMenu/bits.ts`

The `stereo` blueprint lists brown twice: 2 and 5. Eligibility checks each entry independently rather than requiring a cumulative 7. Deduction subtracts 2 first, then can skip the later 5 because the remaining amount is insufficient. A user may complete the blueprint while paying only part of the intended cost.

**Recommendation:** Normalize each blueprint's requirements by color before checking and deducting, or store one aggregated requirement per color.

### 27. `coldplay1` cannot activate its intended theme

**Files:** `components/BlueprintMenu/blueprints.ts`, `components/About/ImageContainer.tsx`, `components/Theme/ThemeContext.tsx`

The object is named `coldplay1`, and its click passes that name to `switchTheme`. ThemeContext supports `coldplay`, not `coldplay1`, so it falls into the default case and switches to dark mode.

**Recommendation:** Align the object/theme identifier or introduce an explicit `themeKey` field.

### 28. Inventory/default help text can cover the overlay

**File:** `components/About/ImageContainer.tsx`

The help `TextBox` has `z-index: 10000`, while the inventory overlay has `z-index: 10`. If the default message is visible when inventory opens, it can render on top of the inventory.

**Recommendation:** Hide contextual room text while inventory is open and standardize layer tokens.

### 29. Whole-scene opacity reduces UI readability

**File:** `components/About/ImageContainer.tsx`

Theme opacity is applied to the entire scene container, affecting text, buttons, inventory, and interactive feedback—not only the background artwork.

**Recommendation:** Apply opacity to a dedicated background layer or pseudo-element, never the full interactive subtree.

## Music

### 30. The page is visibly unfinished and exposes a test control

**File:** `pages/music.tsx`

The page says `Nothing to see here (yet)` and exposes a button that grants test bits. This reads as internal tooling, not a finished portfolio experience.

**Recommendation:** Either hide the route until it has real content, or turn it into an intentional music profile with playlists, current rotation, vinyl collection, or a concise coming-soon presentation. Move bit seeding behind development-only tooling.

### 31. Music content can overflow narrow screens

**File:** `pages/music.tsx`

The main text uses `white-space: nowrap`, and the long test-button label has no constrained width or wrapping strategy.

**Recommendation:** Use responsive max-widths, centered text wrapping, and mobile-safe horizontal padding.

# Theme and visual-system findings

## 32. Two theme colors are invalid CSS

**File:** `components/Theme/themes.tsx`

- `coldplay2.c1` is `#00054`, which is not a valid standard hex color.
- `daft.c4` is `#black`, which is invalid.

**Recommendation:** Correct these values and validate theme objects with a typed color/token test.

## 33. Most flavor/text combinations fail normal-text contrast

**File:** `components/Theme/themes.tsx`

The site frequently renders `c4` text on `c3` backgrounds for tags, controls, inventory sections, and labels. A source-level WCAG contrast calculation found that **14 of 18** theme `c4`/`c3` combinations are below 4.5:1 or invalid.

Examples:

- dark: approximately 2.52:1
- the1975: approximately 1.54:1
- basement: approximately 1.39:1
- glass: approximately 1.03:1
- mckenna: approximately 1.93:1
- daft: invalid because `c4` is invalid

**Recommendation:** Replace generic `c1`–`c4` usage with semantic tokens such as `background`, `surface`, `textPrimary`, `textOnAccent`, `accent`, `border`, and `focusRing`. Generate or validate `textOnAccent` independently for each theme.

## 34. Theme persistence is fragile

**Files:** `components/Theme/ThemeContext.tsx`, `components/BlueprintMenu/bits.ts`, `components/BlueprintMenu/blueprints.ts`

Raw `JSON.parse` calls have no error handling. Corrupted or manually edited localStorage can crash theme, inventory, or blueprint initialization. The entire theme object is persisted rather than a stable theme name, so old schemas can remain indefinitely.

**Recommendation:** Store stable identifiers, validate parsed data, wrap storage access, and recover to safe defaults.

## 35. Generic color-token names are used inconsistently

`c1`–`c4` mean different visual roles across components. This causes contrast mistakes and makes design changes risky.

**Recommendation:** Adopt semantic tokens and document allowed foreground/background pairings.

# P2 — Medium-priority findings

## 36. Missing page metadata and social presentation

**Files:** `pages/_document.tsx`, page components

There is no visible page-level title, description, canonical URL, Open Graph data, Twitter card data, or per-page metadata strategy.

**Recommendation:** Add a shared SEO component and unique metadata for Home, Projects, Explore, and Music.

## 37. Duplicate Google Fonts requests

**File:** `pages/_document.tsx`

Two stylesheet requests include overlapping font families. This creates unnecessary network and stylesheet work.

**Recommendation:** Consolidate to one request or use `next/font`.

## 38. Images are mostly unoptimized

Most content uses plain `<img>` without responsive `srcset`, sizes, lazy-loading policy, dimensions, or Next.js optimization. The repository is asset-heavy, making this important for mobile performance and layout stability.

**Recommendation:** Use `next/image` where practical, define dimensions/aspect ratios, lazy-load below-the-fold project images, and compress large assets.

## 39. Focus-visible styles are absent

Buttons and links generally define hover states but not visible keyboard focus states.

**Recommendation:** Add a consistent, high-contrast `:focus-visible` ring that works across every theme.

## 40. Global `user-select: none` blocks useful copying

**Files:** page containers

Users cannot easily copy project descriptions, titles, credentials, or contact-relevant text.

**Recommendation:** Limit selection blocking to decorative/drag surfaces only.

## 41. Hidden overflow masks layout defects

**File:** `styles/globals.css`

Global `overflow-x: hidden` suppresses evidence of elements exceeding the viewport. This can clip content and focus rings.

**Recommendation:** Fix local overflow and avoid global suppression unless it is proven necessary.

## 42. Global minimum height can harm short/landscape layouts

**File:** `styles/globals.css`

`min-height: 500px` on both `html` and `body` conflicts with small mobile landscape and embedded browser heights.

**Recommendation:** Remove the hard minimum and make individual experiences responsive to short viewports.

## 43. Project card keys use array index

**File:** `pages/projects.tsx`

Filtered/reordered cards use `index` as the React key, which can cause state/identity errors as filters change.

**Recommendation:** Use a stable project ID or slug.

## 44. Several controls lack explicit disabled-state communication

Blueprint and series controls depend mainly on visual styling. Some disabled controls retain low-contrast styling and do not explain why an action is unavailable.

**Recommendation:** Add disabled styling, accessible descriptions, and boundary disabling for previous/next series buttons.

## 45. Content hierarchy and language need a polish pass

Examples include internal/debug language, `Nothing to see here (yet)`, `Masters Degree Receiver`, generic image alt text, and an empty Catfish description.

**Recommendation:** Perform a dedicated content audit focused on recruiter clarity, consistent tense, concise project outcomes, and intentional empty states.

# P3 — Cleanup and maintainability

## 46. Dead or unused state/imports

Examples include unused `isMobileView`, unused `stash` presentation state on Music, and unused imports in inventory components.

**Recommendation:** Remove dead code and enable lint rules for unused imports/variables.

## 47. Repeated style values need shared primitives

Typography, shadows, radii, z-indexes, and transitions are repeated across components.

**Recommendation:** Add shared design tokens and small primitives for buttons, cards, overlays, focus rings, and mono text.

## 48. Theme/object identifiers are implicit strings

The app depends on exact string matches between blueprint names, positions, images, and themes.

**Recommendation:** Introduce typed IDs and explicit fields such as `id`, `positionKey`, and `themeKey`.

# Page-by-page recommended redesign targets

## Home

- Preserve the album-player concept.
- Make the carousel content accurate at every viewport.
- Add a short professional value proposition and clear calls to Projects/contact.
- Reduce footer motion and stop DOM accumulation.
- Ensure the player fits 320px width and short landscape heights.

## Projects

- Use real route navigation and shareable project URLs/slugs.
- Make cards keyboard-operable.
- Add an accessible modal or dedicated project detail pages.
- Improve top spacing beneath the vinyl navigation.
- Add outcomes, responsibilities, and clearer link availability.

## Explore

- Keep the room concept, but provide visible onboarding and native/touch controls.
- Make every collectible object a semantic interactive control.
- Replace hover-only information.
- Redesign inventory as tabs on mobile.
- Fix crafting aggregation and explicit theme keys.

## Music

- Remove the public test-bit utility.
- Either ship a purposeful music experience or temporarily remove the navigation item.
- Make the page contribute to the portfolio narrative rather than appear incomplete.

# Suggested implementation sequence

## Phase 1 — Reliability

1. Restore real Next.js routing.
2. Fix malformed Header CSS.
3. Add CI build/typecheck/lint.
4. Correct invalid colors and the `coldplay1` theme mapping.
5. Aggregate blueprint requirements before validation/deduction.

## Phase 2 — Accessibility

1. Convert clickable divs/images/anchors-without-href to buttons or links.
2. Add focus-visible styles and landmarks.
3. Replace modal behavior with an accessible dialog.
4. Add reduced-motion support.
5. Replace hover-only information with focus/tap-accessible patterns.

## Phase 3 — Responsive UI

1. Redesign mobile navigation.
2. Fix Home player sizing and short-height layouts.
3. Add responsive Projects spacing and modal sizing.
4. Convert inventory to tabs/one-column mobile layout.
5. Prevent text/help overlays from covering dialogs.

## Phase 4 — Design system and polish

1. Replace `c1`–`c4` with semantic tokens.
2. Validate all theme contrast combinations.
3. Consolidate typography, layers, spacing, and component primitives.
4. Add metadata and image optimization.
5. Replace unfinished/debug content and run a final copy review.
